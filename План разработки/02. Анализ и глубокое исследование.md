## 1. Что уже есть внутри проекта (переиспользовать, не искать заново)

### 1.1 Ключи и сервисы, уже зарегистрированные под генерацию контента

В мастер-файле `Project settings/API Keys.txt` есть отдельный раздел **«Контент-завод»**, который
уже группирует именно те сервисы, что нужны Агенту 4 — это не случайность, пользователь заранее
закладывал их сюда:

| Сервис | Статус (по файлу ключей) | Роль для Агента 4 |
|---|---|---|
| **Runway** | Зарегистрирован, ключ есть, нужна оплата | Видео-агрегатор — см. §2.7, лучший кандидат на единую точку входа для видео |
| **Midjourney / DALL-E** | Зарегистрирован, нужна оплата | Изображения — но см. важную оговорку в §2.3.5: у Midjourney нет открытого self-serve API |
| **Replicate** | Зарегистрирован и работает (`r8_...`) | Хостинг открытых моделей (Flux и др.) — уже готовый ключ, не требует поиска альтернатив |
| **ElevenLabs** | Зарегистрирован, бесплатный тариф (вероятно, нужен платный) | Голос/TTS, клонирование голоса под бренд |

**Важно:** Runway у же зарегистрирован и по внешнему исследованию (§2.2) оказывается не просто одним
видео-инструментом, а **агрегатором** — через один API и единую систему кредитов даёт доступ к
собственным моделям (Gen-4.5, Gen-4 Turbo), Google Veo 3.1, Google Gemini Image3 Pro и OpenAI
GPT-Image-2. Это значит, что уже оплаченный/регистрируемый ключ Runway потенциально закрывает и
видео, и часть изображений без отдельной регистрации в Google/OpenAI — стоит проверить в первую
очередь на этапе Архитектуры, прежде чем регистрироваться в других сервисах.

### 1.2 Готовая инфраструктура для анализа виральности — уже спроектирована Агентом 2

Ключевая находка: Агенту 4 **не нужно с нуля изобретать анализ виральных механик** — Агент 2 уже
закладывал под это поля в своей JSON-схеме результата парсинга видео (см. `Intelligence agent/План
разработки/Контент завод - Поиск + Анализ + Парсинг + Создание контента 2 (Codex ver).md`, раздел 8,
и подтверждено в `Deep parsing agent/План разработки/2. Анализ и глубокое исследование.md`, шаг 5):

```json
"combined_analysis": {
  "core_meaning": "Главный смысл видео",
  "hooks": [],
  "triggers": [],
  "offers": [],
  "viral_reasons": [],
  "content_ideas": []
}
```

Финальный LLM-проход Агента 2 (Timeline Merger, шаг 5) уже собирает эти поля «под нужды Агента 3/4»
— формулировка из самого документа Агента 2. То есть цепочка Агент 1 → Агент 2 → Агент 3 уже
доставляет Агенту 4 не сырое видео, а разобранные хук/триггер/оффер/причину виральности/идеи
контента по конкретному просмотренному примеру. Агенту 4 останется читать эти поля через
`analysis_detail` (см. «01. Идея», раздел про контракт Агента 3) и опираться на них при генерации —
не пересобирать анализ виральности своими средствами.

### 1.3 Стоимостная философия — уже согласована во всей цепочке

Каскад «дешёвый инструмент → дорогой, эскалация только при необходимости» — это не изобретение для
Агента 4, а сквозной принцип, уже применённый Агентом 2 (парсинг) и заложенный Агентом 3 (порог
$0.10 на автоматическое решение, выше — вопрос пользователю). Внешнее исследование (раздел 2 ниже)
показывает, что тот же паттерн — стандартная индустриальная практика и для генерации контента в 2026
году, что подтверждает: Агенту 4 не нужно придумывать собственную философию стоимости, только
применить уже проверенный в проекте паттерн к новому домену (генерация вместо парсинга).

### 2.4 Механика виральности — прикладные фреймворки (для промптинга и валидации сценариев)

- **Hook–Retention–Payoff** (специализация AIDA под короткий формат): хук 0–3 сек → удержание/ценность
  →払 payoff/CTA. Согласуется с полями `hooks/triggers/offers/viral_reasons` уже из схемы Агента 2
  (раздел 1.2 выше) — не нужно вводить новую терминологию, можно напрямую мапить на эти поля.
- **Hook–Story–Offer** — вариант из прямого маркетинга, ближе к продающему контенту.
- **Правило первых 3 секунд** — самый резкий обрыв удержания зрителя происходит в первые секунды;
  агент должен явно требовать сильный хук в первом сегменте сценария/видео.
- **Pattern interrupts** — смена ракурса/оверлея/звука каждые 2–4 сек удерживает внимание в среднем
  сегменте ролика.
- **Алгоритмический разбор существующих вирусных роликов** (транскрибация + сегментация сцен + анализ
  аудио + метаданные) — по факту уже частично делает Агент 2 при парсинге (транскрипт, visual_timeline,
  OCR) — Агенту 4 не нужен отдельный пайплайн разбора, только использование уже готового результата
  через Агента 3.

### 2.5 Паттерны оптимизации стоимости на продакшене (индустриальная практика 2026)

- **Каскад дешёвый→дорогой** — черновые варианты на дешёвых моделях (Hailuo-02, `gen4_turbo`,
  Imagen Fast, Deepgram Aura), финальные/hero — на дорогих (Sora 2 Pro, Veo 3.1, ElevenLabs). Уже
  согласуется с философией Агента 2/3 в проекте (раздел 1.3).
- **Кэширование контекста** — переиспользование системного промпта/контекста бренда (guidelines,
  голос, стиль) снижает стоимость LLM-вызовов в разы (пример из исследования: кэшированный ввод
  GPT-5.5 в 10 раз дешевле обычного).
- **Батчинг** — генерация нескольких вариантов за один проход там, где API это поддерживает.
- **Переиспользование ассетов** — не перегенерировать заново то, что уже создано (например,
  клонированный голос бренда создаётся один раз, дальше просто переиспользуется).

## 3. Выводы для следующих этапов

1. **Стек генерации не нужно собирать с нуля** — Runway (уже зарегистрирован) закрывает видео +
   часть изображений одним API; Deepgram/ElevenLabs (уже есть ключи) закрывают голос; Replicate (уже
   работает) — резерв для открытых моделей (Flux и др.). Финальный выбор — предмет этапа «04. ТЗ» /
   стека ресурсов, но искать новые сервисы с нуля не требуется.
2. **Midjourney** из уже зарегистрированных ключей проекта **не годится** для автоматической
   генерации — нет публичного API (только Discord/ручной режим). Учитывать при выборе
   image-генератора.
3. **Анализ виральности не нужно проектировать заново** — использовать готовые поля
   `hooks/triggers/offers/viral_reasons/content_ideas`, которые уже формирует Агент 2 и передаёт
   через Агента 3.
4. **Философия каскада стоимости** переносится из Агента 2/3 без изменений, только применяется к
   генерации вместо парсинга.
5. Точные модели/тарифы фиксировать не раньше этапа «04. ТЗ» — цены и модели в этой сфере меняются
   быстро (несколько раз в год), это исследование — ориентир на момент 2026-07, не жёсткая
   спецификация.
