## 1. Принципы

| Принцип | Как реализуется |
|---|---|
| Не изобретать заново | Транспорт, каскад стоимости, паттерн буфера — переиспользуются из Агента 2/3, не проектируются с нуля |
| Единая точка входа для пользователя | Агент 1 — весь UI/UX, Агент 4 никогда не пишет пользователю напрямую («05. ТЗ», §5.4/9.3) |
| Кто знает — тот и будит | Агент 1 точно знает, кто и что запросил — он инициирует работу Агента 4, а не наоборот (§4) |
| Push = указатель, не данные | И у Агента 1→4, и у Агента 3→4 очередь несёт только идентификатор, содержимое запрашивается отдельно |
| Owned schema | Агент 4 пишет только в свою схему `content_creation_agent`, читает чужие только `.schema(...)`-запросами, никогда не пишет в чужие схемы |
| Дёшево → дорого | Каскад стоимости для генерации, тот же принцип, что у Агента 2 (парсинг) и Агента 3 (эскалация) |

## 4. Вход от Агента 1 — главный триггер (решено 2026-07-10)

**Основной способ, которым Агент 4 узнаёт о работе.** Требует новой доработки Агента 1 (см.
`Доработки для Агентов 1 и 3 (передать).md`, раздел D) — по образцу уже реализованного и проверенного
паттерна Агент 3 → Агент 4.

### 4.1 Redis (быстрый слой, best-effort)

Тот же канал, что уже слушает Агент 4 для Агента 3 — **`notifications:agent4`**, но с другим
`event`, чтобы Агент 4 мог разбирать оба типа сообщений одной подпиской:

```json
{ "event": "wizard_ready", "telegram_id": 123456789, "wizard_hash": "sha256:...", "mode": "content", "timestamp": "ISO-datetime" }
```

### 4.2 Supabase (надёжный слой)

Новая таблица `intelligence_agent.agent4_handoff_queue` (владелец — Агент 1, схема
`intelligence_agent`, по аналогии с тем, что у Агента 3 очередь лежит в схеме отправителя, не
получателя):

```sql
CREATE TABLE intelligence_agent.agent4_handoff_queue (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  telegram_id     BIGINT NOT NULL,
  wizard_hash     TEXT NOT NULL,
  mode            TEXT NOT NULL,          -- 'content' | 'publish'
  attempt_count   INTEGER NOT NULL DEFAULT 0,
  last_attempt_at TIMESTAMPTZ,
  status          TEXT NOT NULL DEFAULT 'pending',
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Запись — с теми же ретраями `[500мс, 2с, 8с]`, что уже проверены в реализации Агента 3
(`agent4Handoff.js`), только на сам INSERT. Redis-публикация — best-effort, без ретраев, ошибка не
блокирует основной поток (тот же принцип). **Агент 1 не обновляет `status` после вставки** — как и
Агент 3, оставляет это Агенту 4.

### 4.3 Получение самого wizard'а

Очередь несёт только указатель (`telegram_id`+`wizard_hash`), не сам wizard. Агент 4, получив
уведомление, читает полный объект:

```
Таблица: intelligence_agent.users
Колонка: settings (jsonb), ключ: telegram_id

settings = {
  mode: "info" | "content" | "publish",
  wizard: { network, content_type, format, style, description },
  moderation_mode: boolean
}
```

Сверяет `wizard_hash` из очереди с `hash(wizard)` из прочитанного объекта — если не совпадает,
значит пользователь успел изменить wizard после публикации события, берётся актуальная версия
(и в этом случае `wizard_hash` в дедуп-таблице тоже должен быть от актуальной версии, см. §7).

## 5. Вход от Агента 3 — опциональное обогащение (реальный контракт, сверено с кодом v0.7.0)

Уже реализованный push-хендофф Агента 3 остаётся в инфраструктуре, но **больше не основной триггер**
— теперь это источник трендовых данных **по требованию**, когда wizard просит опору на тренды (см.
свободный текст шага 5 — сейчас нет отдельного явного переключателя, см. открытый вопрос §10).

### 5.1 Redis / Supabase (без изменений от предыдущей версии документа)

- Redis `notifications:agent4`, `{ event: "digest_ready", run_id, timestamp }` — best-effort, без
  ретраев (`Information analysis agent/Code/src/handoff/agent4Handoff.js:48-52`).
- Supabase `information_analysis_agent.agent4_handoff_queue` (`job_id`/`result_ref` = `run_id`,
  ретраи `[500,2000,8000]`мс на INSERT, статус после вставки не обновляется — Агент 4 сам ведёт учёт).

### 5.2 Как используется теперь

Агент 4 **не подписывается на `digest_ready` как на триггер работы** — вместо этого, когда генерация
запрошена с опорой на тренды, Агент 4 сам инициирует запрос к Агенту 3:

1. `analysis_status()` / последний известный `run_id` (из своей таблицы `inbox_events`, см. §7, куда
   `digest_ready` события всё равно складываются пассивно — на случай если понадобятся) или прямой
   вызов `analysis_digest()` за последним прогоном.
2. `analysis_detail(claim_id)` — для конкретных фактов, релевантных теме wizard-запроса (сопоставление
   по ключевым словам/теме — задача LLM-фильтра внутри Агента 4, не Агента 3).
3. Если данных недостаточно — сырые данные напрямую из `intelligence_agent.search_results` /
   `deep_parsing_agent.parsing_results` (см. «01. Идея», обязательно `.schema(...)`).

Без запроса на тренды этот слой полностью пропускается.

### 5.3 Известное расхождение с документацией Агента 3 (не блокирует Агента 4)

Собственный архитектурный документ Агента 3 (`Information analysis agent/План разработки/7.
Архитектура (Бекенд).md`, §4.1/§6.3) всё ещё описывает старую чистую pull-модель и противоречит
собственному коду — зафиксировано как доработка C в файле доработок, не в скоупе Агента 4.

## 6. Модель данных Агента 4 (схема `content_creation_agent`)

```sql
-- Основная таблица результатов генерации
generated_content (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  telegram_id     bigint NOT NULL,
  wizard_hash     text NOT NULL,
  run_id          uuid,                  -- прогон Агента 3, если использовалось обогащение трендами
  type            text NOT NULL,         -- text | image | video | audio
  r2_url          text,
  social_target   text[],
  status          text NOT NULL DEFAULT 'pending',  -- pending|processing|done|error|published|publish_failed
  publish_report  jsonb,                 -- { instagram: "ok", tiktok: "failed: ..." }
  cost_usd        numeric,
  metadata        jsonb,
  published_at    timestamptz,
  created_at      timestamptz NOT NULL DEFAULT now()
)

-- Дедупликация wizard-запросов: не генерировать дважды один и тот же запрос
processed_wizard_requests (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  telegram_id     bigint NOT NULL,
  wizard_hash     text NOT NULL,
  processed_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (telegram_id, wizard_hash)
)

-- Входящие события от обоих отправителей (Агент 1 и Агент 3), единая точка учёта
inbox_events (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  source          text NOT NULL,         -- 'agent1_wizard' | 'agent3_digest'
  ref             text NOT NULL,         -- wizard_hash (для agent1) или run_id (для agent3)
  telegram_id     bigint,                -- заполнено только для agent1_wizard
  status          text NOT NULL DEFAULT 'pending',  -- pending|processed|skipped
  received_via    text NOT NULL,         -- 'redis' | 'poll'
  received_at     timestamptz NOT NULL DEFAULT now()
)
```

`inbox_events` — общая точка учёта входящих сигналов от обоих источников: не даёт Агенту 4 зависеть
только от Redis pub/sub (теряет сообщения при недоступности) и не тратить лишние вызовы MCP на уже
обработанные события при поллинге обеих очередей (`intelligence_agent.agent4_handoff_queue` и
`information_analysis_agent.agent4_handoff_queue`).

## 8. Нефункциональные детали

### 8.1 Отказоустойчивость
- Redis-уведомления (от обоих источников) — best-effort, не единственный источник истины: поллинг
  обеих Supabase-очередей — обязательный catch-up механизм, не опциональный.
- Файл в R2 пишется **до** отметки в Supabase как `done` — при сбое между шагами перегенерации не
  требуется, только повторная попытка записи метаданных.

### 8.2 Безопасность/доступ
- `service_role` ключ Supabase (уже используется всеми агентами, `BYPASSRLS`) — доступ к чужим схемам
  `intelligence_agent`/`information_analysis_agent` только на чтение (`SELECT`), никаких `UPDATE`/
  `INSERT` в чужие схемы.
- R2 — отдельный bucket-токен, не пересекается с ключами других агентов.

### 8.3 Стоимость
Оценка стоимости перед дорогим этапом каскада — до вызова платной модели. Порог автоматического
продолжения без подтверждения пользователя — **$0.50** (выше, чем $0.10 у Агента 3, т.к. генерация
видео дороже парсинга по своей природе).
