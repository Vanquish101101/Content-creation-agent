# 05. Техническое задание — Агент создания контента (Агент 4)

## 1. Назначение и область действия

Агент 4 — четвёртое, завершающее звено в цепочке из 4 специализированных агентов. Отвечает за
генерацию контента (текст, фото, видео, аудио) на основе запроса пользователя и данных, синтезированных
Агентом 3, и опционально — за публикацию готового контента в соцсети.

**В зоне ответственности:** генерация контента любого типа, опциональная опора на трендовые механики
(хук/триггер/оффер), контроль и оптимизация стоимости генерации, хранение готовых файлов, доставка
пользователю, опциональная публикация.

**Вне зоны ответственности:** поиск источников (Агент 1), парсинг/транскрибация (Агент 2), семантический
синтез/дедупликация/контроль качества входных данных (Агент 3) — Агент 4 потребляет уже готовый, синтезированный
результат, не занимается его добыванием.

Источники решений: `01. Идея.md`, `02. Анализ и глубокое исследование.md`, `04. Брейншторм.md` этого
проекта, плюс восстановленный дизайн из `Marketing agency/План разработки/4. Архитектура.md`
(до-разделения мастер-проект).

## 2. Цели и задачи

### 2.1 Главная цель

Выполнять функцию контент-завода: по запросу пользователя (напрямую или через данные Агентов 1–3)
производить готовый к использованию контент нужного типа, формата и стиля, с контролируемой
стоимостью, без потери качества.

### 2.2 Задачи агента

1. Принимать готовые синтезированные данные от Агента 3 (push-уведомление + чтение дайджеста).
2. Генерировать контент по каскаду «дёшево → дорого» в зависимости от типа (текст/фото/видео/аудио).
3. При запросе пользователя — опираться на трендовые механики (`hooks/triggers/offers/viral_reasons`,
   уже подготовленные Агентом 2/3), создавая контент в похожей механике, но на материале пользователя.
   Без такого запроса — генерировать без оглядки на тренды.
4. Адаптировать контент под выбранную платформу (формат/соотношение сторон/стиль).
5. Сохранять готовые файлы в Cloudflare R2, метаданные — в Supabase.
6. Отдавать пользователю короткий отчёт + превью/ссылки на результат (не пачкой файлов в чат).
7. При выборе пользователем режима «создать и опубликовать» — публиковать через PostMyPost и
   присылать отчёт об успехе/неудаче по каждой выбранной соцсети.
8. Учитывать и логировать стоимость генерации, по аналогии с `/costs` у Агента 1.

### 2.3 Критерии успеха

- Пользователь получает результат нужного типа/формата за приемлемое время без ручного вмешательства.
- Стоимость генерации остаётся в рамках каскадной экономии (дешёвые модели — черновики/массовые
  варианты, дорогие — только финальные/hero-ассеты).
- Публикация (если запрошена) либо проходит успешно, либо даёт понятный отчёт о причине неудачи по
  каждой площадке отдельно.
- Ни один сгенерированный файл не теряется между генерацией и доставкой пользователю (буфер/хранилище
  надёжны).

## 3. Режимы вызова

Точка входа — Telegram-бот Агента 1 (см. §9.1 «Интеграция»). Пользователь выбирает один из трёх
режимов при старте диалога:

| Режим | Агенты | Публикация |
|---|---|---|
| 🔍 Получить информацию | 1+2+3 | — |
| 🎬 Создать контент | 1+2+3+4 | Нет |
| 🚀 Создать и опубликовать | 1+2+3+4 | Да (PostMyPost) |

Для режимов 2/3 запускается диалоговый wizard (реализуется на стороне Агента 1, доработка описана в
`Доработки для Агентов 1 и 3 (передать).md`): соцсеть(и) → тип контента → формат → стиль → задача
(свободный текст). Wizard формирует структурированный запрос, который транспортируется до Агента 4
вместе с/после данных Агента 3.

## 4. Функциональные требования

### 4.1 Генерация контента по типам

Каскад «дёшево → дорого», единый принцип для всех типов (по аналогии с каскадом Агента 2):

| Тип | Черновик | Финал | Hero/премиум |
|---|---|---|---|
| Видео | MiniMax Hailuo-02 / Runway `gen4_turbo` | OpenAI Sora 2 / Runway `gen4.5` | Veo 3.1 / Sora 2 Pro |
| Изображения | Imagen 4 Fast / `gen4_image_turbo` | Ideogram 4.0 Turbo / Imagen 4 Standard | Imagen 4 Ultra / Ideogram Quality |
| Голос | Deepgram Aura | ElevenLabs (клонированный голос) | ElevenLabs Pro |
| Текст | дешёвая LLM (черновики/варианты) | основная LLM (финальный пост) | — |

Runway — приоритетная точка входа для видео и части изображений (агрегирует несколько моделей одним
API, уже зарегистрирован). Midjourney — исключён из автоматизации (нет публичного API).

### 4.2 Опора на тренды (опционально)

Если пользователь явно запросил трендовый контент — Агент 4 читает поля `combined_analysis.hooks/
triggers/offers/viral_reasons/content_ideas` из данных, пришедших через Агента 3 (эти поля уже
формирует Агент 2 при парсинге референсного контента), и использует их как структурный шаблон
(Hook–Retention–Payoff) для генерации: сохраняется механика/структура референса, не его буквальное
содержание. Без явного запроса — генерация идёт по свободному описанию задачи пользователя, без
привязки к какому-либо референсу.

### 4.3 Публикация (опционально)

Только в режиме «Создать и опубликовать». Публикация через PostMyPost в выбранные пользователем
соцсети. После попытки — отчёт по каждой площадке отдельно: успех/неудача и причина при неудаче.
Режим модерации (автопилот vs подтверждение перед публикацией) — переключатель, значение по
умолчанию для MVP уточняется в §12 (открытые вопросы).

### 4.4 Хранение

Все сгенерированные файлы — в Cloudflare R2. Supabase (`content_creation_agent`) хранит только
ссылку + метаданные (см. §7). Политика хранения: N последних прогонов держатся в R2 даже после
доставки/публикации (подстраховка), точное N — открытый вопрос (§12).

### 4.5 Доставка пользователю

В Telegram-чат приходит короткий отчёт + превью/ссылки на все файлы, с пометками (соцсеть/тип/
описание/размер) — не пачка файлов. Пользователь может запросить конкретный файл или всё целиком
отдельной командой/кнопкой.

### 4.6 Оценка и контроль стоимости

Перед запуском дорогого этапа каскада — оценка ожидаемой стоимости. Логирование фактических трат по
каждому запросу генерации, доступное пользователю (аналог `/costs` у Агента 1). Порог автоматического
продолжения без подтверждения пользователя — по аналогии с порогом Агента 3 ($0.10) — уточняется на
этапе Архитектуры (заимствуется, но объёмы генерации контента дороже парсинга/анализа — порог может
отличаться).

## 5. Интерфейсы

### 5.1 Вход — от Агента 1 (push, главный триггер работы Агента 4, решено 2026-07-10)

**Основной способ, которым Агент 4 узнаёт о работе.** Агент 1, сразу по завершении диалогового
wizard'а (соцсеть/тип/формат/стиль/задача), сам инициирует передачу — тем же уже проверенным
паттерном, что использует Агент 3 → Агент 4 (см. §5.2):

1. Redis `notifications:agent4`, событие `wizard_ready` — быстрое уведомление.
2. Supabase `intelligence_agent.agent4_handoff_queue` — надёжная запись (новая таблица, доработка
   Агента 1, см. `Доработки для Агентов 1 и 3 (передать).md`, раздел D).
3. При сбое записи — in-process буфер на стороне Агента 1 с ретраями (тот же принцип, что уже
   реализован у Агента 3).

Очередь несёт только указатель (`telegram_id`+`wizard_hash`+`mode`), не сам wizard — Агент 4,
получив уведомление, читает полный объект `intelligence_agent.users.settings.wizard` напрямую
(cross-schema, `.schema('intelligence_agent')`).

**Почему так, а не наоборот (Агент 4 сам вычисляет, что для него есть работа):** более ранний вариант
предполагал, что Агент 4 отслеживает прогоны Агента 3 и постфактум сопоставляет их с
пользовательскими запросами — оказалось хрупко (прогоны Агента 3 пакетные, не привязаны к
конкретному запросу, нет гарантии что нужный `telegram_id` попадёт в ближайший batch) и добавляет
задержку. Агент 1 точно знает, кто и что запросил, в момент запроса — логичнее, чтобы он и будил
Агента 4 напрямую.

### 5.2 Вход — от Агента 3 (опциональное обогащение трендовыми данными)

Существующий push-хендофф Агента 3 (Redis `notifications:agent4`, событие `digest_ready` + Supabase
`information_analysis_agent.agent4_handoff_queue`, см. `Доработки для Агентов 1 и 3 (передать).md`,
раздел B) остаётся в инфраструктуре, но перестаёт быть основным триггером работы — теперь это
источник данных **по требованию**, а не сигнал "проснись".

Используется только когда пользователь в wizard'е (свободный текст задачи) просит опору на тренды:
Агент 4 запрашивает у Агента 3 актуальный дайджест (`analysis_digest`/`analysis_detail`, MCP порт
7302) и ищет факты, связанные с темой запроса. Если дайджеста недостаточно — сырые данные напрямую из
`intelligence_agent.search_results` / `deep_parsing_agent.parsing_results` (см. «01. Идея», раздел про
cross-schema чтение — обязательно `.schema(...)`, иначе повторится уже дважды случавшийся баг). Без
запроса на тренды — этот шаг полностью пропускается, Агент 4 генерирует по одному только wizard'у.

### 5.3 Выход — MCP-сервер Агента 4

По аналогии с Агентом 2/3, Агент 4 поднимает собственный MCP-сервер (порт — вероятно 7303, не
зафиксировано) для статуса и управления генерацией:

- `content_generate({ request })` — запуск генерации.
- `content_status({ job_id })` — статус текущей генерации.
- `content_result({ job_id })` — ссылки/метаданные готового результата.

Точная сигнатура — предмет этапа Архитектуры.

### 5.4 Выход — Telegram (решено 2026-07-09)

**Агент 4 не использует собственный Telegram-токен.** В отличие от Агента 3 (у которого есть прямой
канал уведомлений своим токеном), финальный отчёт с превью/ссылками отправляется пользователю
**через Агента 1** — он остаётся единственной точкой входа и управления для пользователя, все
остальные агенты работают только «за кулисами» и отдают результат обратно по цепочке, а не открывают
собственные каналы к пользователю. Это согласуется с общим «видением единого бота» (см. память
проекта): весь UI/UX и вся переписка с пользователем идут через Агента 1, независимо от того, какие
агенты фактически обрабатывали задачу.

Технически это значит: Агент 4, закончив генерацию, кладёт отчёт+ссылки туда же, откуда Агент 1
сможет их забрать (тот же push-паттерн Redis/Supabase, что и на входе, см. §5.1 — теперь в обратную
сторону, Агент 4 → Агент 1). Точный транспорт этого обратного канала — предмет этапа Архитектуры.

### 5.5 Выход — PostMyPost

Вызов API PostMyPost для публикации (ключ уже в проекте). Формат вызова и маппинг типов контента на
параметры PostMyPost — предмет этапа Архитектуры.

## 7. Модель данных (Supabase, схема `content_creation_agent`)

```
Таблица: generated_content
id              uuid          PK
run_id          uuid          FK на прогон Агента 3 (information_analysis_agent.runs)
request_id      uuid          NULL если запуск не из wizard'а
type            text          text | image | video | audio
r2_url          text          Ссылка на файл в Cloudflare R2
social_target   text[]        Соцсети, для которых предназначен (если применимо)
status          enum          pending | processing | done | error | published | publish_failed
publish_report  jsonb         { instagram: "ok", tiktok: "failed: причина", ... }
cost_usd        numeric       Фактическая стоимость генерации
metadata        jsonb         формат, стиль, размер, использованные инструменты
published_at    timestamptz
created_at      timestamptz
```

Точные типы/ограничения — финализировать на этапе Архитектуры. Таблица `agent4_handoff_queue`
физически живёт в схеме `information_analysis_agent` (владелец — Агент 3, см. §5.1), не здесь.

## 10. Нефункциональные требования

### 10.1 Производительность

Генерация текста/изображений — секунды-десятки секунд. Видео — минуты (зависит от каскада и модели).
Пользователь должен получать индикацию прогресса, если генерация превышает ~10 секунд (по аналогии с
индикатором генерации у мастер-проекта).

### 10.2 Надёжность

Ни один результат генерации не должен теряться между созданием и доставкой — файл сразу пишется в R2,
ссылка сразу в Supabase, доставка пользователю — отдельный шаг, повторяемый при сбое (сам файл уже
сохранён, не нужно перегенерировать).

### 10.3 Стоимость

Каскадная экономия (см. §4.1, §4.6) — обязательный принцип, не опция. Средняя стоимость на единицу
контента — логируется и доступна для анализа (`cost_usd` в §7).

## 11. MVP — первая версия

### 11.1 Что входит

- Все 4 типа контента (текст/фото/видео/аудио) — см. решение в «04. Брейншторм», §1.
- Каскад дёшево→дорого по каждому типу.
- Push-приём задачи от Агента 1 (после доработки Агента 1, см. §5.1) + опциональное обогащение
  данными Агента 3 по требованию (§5.2).
- Хранение в R2 + метаданные в Supabase.
- Доставка отчётом+ссылками в Telegram.
- Публикация через PostMyPost (режим 3), с отчётом по соцсетям.
- Базовое логирование стоимости.

### 11.2 Что откладывается на v2

- Режим «Модерация» с интерактивным подтверждением перед публикацией (если для MVP выбран автопилот —
  см. открытый вопрос §12).
- Тонкая настройка retention-политики R2 (сначала — простое фиксированное N).
- Расширенная аналитика по эффективности контента (что "выстрелило" после публикации) — требует
  данных постфактум от соцсетей, отдельная задача.
- Собственный LangGraph-оркестратор для сложных многошаговых цепочек генерации (если простого роутера
  окажется недостаточно).
