# 01. Идея — Агент создания контента (Агент 4)

## Суть идеи

Агент 4 — последнее, завершающее звено в цепочке из 4 специализированных агентов. Его задача —
создание контента любой сложности (посты, тексты, фото, видео, аудио) на основании описания и
запроса пользователя, с учётом его предпочтений, **и без обязательной опоры на текущие тренды**.

При явном указании пользователя агент может опираться на актуальные тренды — используя данные,
которые уже нашли и подготовили Агенты 1–3 (что сейчас популярно и "выстрелило"), — и создавать
контент, похожий по механике/структуре на успешный трендовый контент, но на тему/материал
пользователя. Пользователь также может явно запросить контент, который НЕ опирается на трендовый
формат — это его выбор, а не обязательное поведение агента.

По сути Агент 4 выполняет функцию **контент-завода**: массовое производство постов, текстов, фото,
видео и аудио с постоянной оценкой стоимости и стремлением удешевить генерацию, не теряя в качестве
(тот же принцип каскада "дешёвый → дорогой инструмент", что уже использует Агент 2 при парсинге).

## Место в цепочке агентов

```
Агент 1 (Разведчик)  →  Агент 2 (Глубокий парсер)  →  Агент 3 (Анализатор)  →  Агент 4 (этот проект)
   поиск источников        разбор видео/аудио/            семантический           создание контента
   и быстрые факты          документов/изображений          синтез, дедуп,
                                                              противоречия
```

Все четыре агента — независимые проекты/репозитории в `Marketing agency Project`, связанные через
общий Supabase-проект `Marketing agency` (id `wklecdbujgdwnbmfmggi`, отдельная схема на каждого
агента — для Агента 4 уже зарезервирована пустая схема `content_creation_agent`) и через MCP —
каждый агент может выступать MCP-сервером для следующего.

Не решено окончательно (открытый вопрос, см. ниже), будет ли Агент 4 читать выход Агента 3
напрямую, или вызовы пойдут через единого пользовательского бота Агента 1.

## Что реально отдаёт Агент 3 — контракт входа (проверено по коду, не только по ТЗ)

Агент 3 — read-only MCP-сервер (Streamable HTTP, порт 7302, путь `/mcp`, контейнер
`information-analysis-agent`, сеть `marketing-agency-net`). Три инструмента:

- **`analysis_digest({ run_id? })`** — дайджест последнего (или указанного) прогона: список
  `facts` (связное утверждение + `confidence.level/sources_count/reach_estimate` + `detail_ref`),
  список `contradictions`, `meta` (items_processed/escalations/cost_usd/duration_sec).
  Нюансы: `meta.duration_sec` пока всегда `null`; `reach_estimate` сейчас считается только для
  YouTube-источников (views+likes), для остальных — 0; если дайджест не сохранился — возвращается
  пустая форма, не ошибка.
- **`analysis_detail({ claim_id })`** — полный разбор факта: `statement`, `sources[]`
  (`type/ref/excerpt/confidence`), `reasoning`, `history`. Нюансы: `sources[].excerpt` **всегда
  `null`** (реальной цитаты в дайджесте нет), `history` **всегда пустой массив** — если Агенту 4
  нужна точная цитата/фрагмент исходника, дайджеста недостаточно, нужно идти за сырыми данными
  (см. ниже). `sources[].ref` = `raw_job_id` — ключ для перехода к исходной записи.
- **`analysis_status()`** — `last_run_at/status/items_processed/cost_usd/pending_user_decisions[]`
  — полностью реальные данные, без заглушек.

## Если дайджеста недостаточно — доступ к сырым данным

Агент 3 намеренно не хранит сырой контент, только subject/predicate/object-факты. Через
`analysis_detail.sources[].ref` (= `raw_job_id`) можно дойти до исходника:

- **От Агента 1** (`intelligence_agent.search_results`, `job_id` = `raw_job_id`): поле `result.raw`
  содержит `perplexity.summary`+`citations`, полный список YouTube-видео
  (`title/views/likes/url/channel/description`), полный список Firecrawl-страниц (`title/url/content`).
- **От Агента 2** (`deep_parsing_agent.parsing_results`, `job_id` = `raw_job_id`): поле
  `result_json` — полный результат разбора модуля (полный транскрипт видео/аудио, полный OCR-текст,
  описание визуального ряда). Хранится постоянно в Supabase, не только в Redis.

**Важно при реализации:** обе таблицы физически живут в схемах `intelligence_agent`/
`deep_parsing_agent`, не в схеме `content_creation_agent` — обязательно использовать
`.schema('intelligence_agent')`/`.schema('deep_parsing_agent')` при чтении через Supabase-клиент
(та же ошибка уже дважды случалась у Агента 3, легко повторить заново, если не учесть сразу).

**Чего у Агента 2 нет нигде постоянно:** метаданные стоимости/инструментов конкретного
parsing-job'а (`cost_usd`/`tools_used`/`duration_sec`) — если это важно для Агента 4, потребуется
доработка на стороне Агента 2.

## Задачи агента (черновой список)

1. **Генерация контента по запросу** — посты, статьи/тексты, фото, видео, Reels/Shorts/TikTok,
   аудио — на основе описания и предпочтений пользователя.
2. **Опциональная опора на тренды** — по явному запросу пользователя: анализ похожего
   "выстрелившего" контента (через данные Агентов 1–3) и создание контента в похожей механике/
   структуре, но на материале пользователя.
3. **Адаптация под платформу** — формат/длина/стиль под конкретную площадку публикации.
4. **Контроль и оптимизация стоимости** — каскад "дешёвый → дорогой инструмент" (по аналогии с
   Агентом 2), логирование трат, попытка удешевить генерацию без потери качества.
5. **Публикация** *(черновая идея, не решено)* — в ключах проекта уже есть PostMyPost
   (см. `Project settings/API Keys.txt`) — потенциальный инструмент автопубликации, требует
   отдельного решения на этапе стека/архитектуры.
