## ✅ Живая сквозная проверка Слайсов 1+2 (2026-07-10) — прогнано по-настоящему, не на фейках

Заведён реальный `Code/.env` (Supabase/Redis — общие для всех 4 агентов, ключ переиспользован из

(`telegram_id 999000001`, заведомо не пересекается с реальным пользователем `1064521326`) с
wizard-настройками и записью в `intelligence_agent.agent4_handoff_queue`, агент запущен по-настоящему
(`node src/index.js`). Результат: поллинг забрал задачу → кросс-схемное чтение wizard'а →
дедуп-проверка → каскад текста (реальный вызов OpenRouter, `claude-haiku-4-5`, эскалация не
понадобилась) → запись в `generated_content` со статусом `done`, реальная стоимость **$0.001622**,
готовый текст поста на русском. Тестовые данные (пользователь, очередь, результат) удалены после
проверки, реальный пользователь не тронут.

**Найдено и исправлено живой проверкой — два реальных бага, оба не ловились тестами на фейках:**

1. **Отсутствовал `GRANT USAGE ON SCHEMA content_creation_agent`** — миграция 001 давала `GRANT` только
   на таблицы, но в Postgres это не работает без `GRANT USAGE ON SCHEMA` в дополнение (у остальных 3
   агентов это было сделано, здесь — забыто). Ошибка: `permission denied for schema
   content_creation_agent`. Исправлено новой миграцией `002_schema_grant.sql`, применена к живой БД.
2. **Реальный `supabase-js` `.single()` на пустом результате возвращает ошибку `PGRST116`
   ("Cannot coerce the result to a single JSON object"), а не `{data:null, error:null}`**, как
   ошибочно предполагал фейковый `makeFakeDb` при написании тестов Слайса 1. Ломало
   `isWizardProcessed` (упало бы на первом же новом, ещё не обработанном wizard-запросе) и
   `fetchWizardSettings` (упало бы на несуществующем `telegram_id`). Добавлен общий хелпер
   `src/db/errors.js` (`isNotFoundError`), тесты в обоих местах переписаны под реальное поведение
   PostgREST, добавлен новый тест `db/errors.test.js`.
