## 1. MVP-скоп (решено)

Весь контент-завод сразу: текст/посты, фото, видео, аудио — без поэтапного урезания по типам контента.
Не начинаем с "только текст", как это делают некоторые контент-агенты на старте — риск выше, но не
требует повторного пересмотра архитектуры позже.

## 2. Точка входа и режимы диалога (доработка Агента 1)

Восстановлена логика из мастер-проекта, не перенесённая в текущий Агент 1. Пользователь в начале
диалога выбирает один из **трёх режимов** (было два — «получить информацию» / «создать контент»):

1. 🔍 **Получить информацию** — Агенты 1+2+3, без Агента 4 (как сейчас)
2. 🎬 **Создать контент** — Агенты 1+2+3+4, без публикации
3. 🚀 **Создать и опубликовать** — Агенты 1+2+3+4 + публикация в соцсети

Для режимов 2/3 — диалоговый wizard (источник: `Marketing agency/План разработки/4. Архитектура.md`,
Модуль 2): 1) выбор соцсети(ей) → 2) тип контента (пост/видео/фото/аудио/Reels) → 3) формат/соотношение
сторон → 4) стиль подачи → 5) свободное описание задачи. Для режима 3 добавляется выбор целевых
соцсетей для публикации (мультивыбор — Instagram/YouTube/TikTok/Telegram/VK/WhatsApp/Одноклассники).

## 3. Передача данных Агент 3 → Агент 4 (доработка Агента 3)

Текущая архитектура Агента 3 (`Information analysis agent/План разработки/7. Архитектура (Бекенд).md`,
§6.3) — чистая pull-модель: «Агент 4 просто не получит [данные], если сам недоступен», буфера на
стороне Агента 3 нет. Решено доработать это до гибрида, по аналогии с уже существующим паттерном
Агент 2 → Агент 3 (`agent3_handoff_queue`):

- Агент 3, закончив прогон (GlobalSynthesis) и сохранив дайджест в Supabase, сам инициирует передачу
  Агенту 4 — сначала пишет в **Redis** (быстрое уведомление о новой работе, аналог
  `queue:deep-parsing:jobs`) и **Supabase** (`information_analysis_agent.agent4_handoff_queue` —
  новая таблица, по образцу `deep_parsing_agent.agent3_handoff_queue`: `id/job_id/result_ref/
  attempt_count/last_attempt_at/status`).
- Если запись в Redis/Supabase не удалась (редкий случай — сбой самого хранилища) — **буфер в памяти
  процесса Агента 3** с ретраями и бэкоффом, чтобы результат прогона не терялся.
- Агент 4 получает уведомление о готовой работе, забирает данные либо напрямую из Supabase, либо через
  уже существующий MCP Агента 3 (`analysis_digest`/`analysis_detail`) для деталей по конкретному факту.

Полная техническая формулировка — в `Доработки для Агентов 1 и 3 (передать).md`.

## 6. Хранилище — Cloudflare R2 (решено)

Все сгенерированные файлы — **Cloudflare R2** (аккаунт уже зарегистрирован в проекте, egress
бесплатный всегда — критично для контента, который многократно просматривают/скачивают/публикуют).
Supabase (схема `content_creation_agent`) хранит только ссылки + метаданные — по аналогии с таблицей
`generated_content` мастер-проекта:

```
id, type (text|image|video|audio), r2_url, social_target, status (pending|approved|published|rejected),
published_at, metadata (jsonb: соцсеть/формат/стиль/размер), created_at
```

**Почему не Supabase Storage:** Free-план текущего Supabase-проекта даёт всего 1 ГБ storage / 5 ГБ
egress в месяц — **общие на все 4 агента**, видео исчерпает их за считанные дни. R2 даёт 10 ГБ бесплатно
и $0.015/ГБ дальше, без платы за egress вообще.

Политика хранения: N последних прогонов держим "про запас" в R2 даже после доставки пользователю или
публикации (подстраховка на случай потери/запроса повтора) — точное число N и срок жизни — открытый
вопрос перед ТЗ.

### Справка: почему R2, а не Supabase Storage (цифры проверены напрямую, 2026-07-09)

**Supabase Storage** — уже встроен в проект `Marketing agency` (тот же проект, которым пользуются все
4 агента для БД). Файл → bucket → URL, в Postgres пишется только ссылка+метаданные. Аккаунт
(`Vanquish Org`, проверено через Supabase MCP) сейчас на **Free-плане**:

| Параметр | Free (сейчас) | Pro ($25/мес) |
|---|---|---|
| Storage (место под файлы) | **1 GB всего** | 100 GB, дальше $0.021/GB |
| Egress (скачивание/просмотр) | **5 GB/месяц** | 250 GB, дальше $0.09/GB |
| Database (сами таблицы) | 500 MB | 8 GB |

Лимит общий на весь проект, не отдельно на Агента 4 — видео исчерпает 1 ГБ за считанные дни.

**Cloudflare R2** — S3-совместимое объектное хранилище, аккаунт уже зарегистрирован в проекте (только
R2-специфичная настройка ещё не сделана). Актуальные тарифы (проверено на
developers.cloudflare.com/r2/pricing):

| Параметр | Бесплатно | Платно |
|---|---|---|
| Storage | 10 GB-месяц | $0.015/GB-месяц |
| Запись/загрузка (Class A) | 1 млн операций/мес | $4.50/млн |
| Чтение/просмотр (Class B) | 10 млн операций/мес | $0.36/млн |
| **Скачивание (egress)** | **Всегда бесплатно, без лимита** | — |
| Удаление файла | Всегда бесплатно | — |

Egress у R2 бесплатен всегда — принципиально важно для контент-завода, где один файл генерируется
один раз, а смотрят/скачивают/публикуют его много раз (именно за повторные просмотры Supabase/S3
берут деньги). Пример из документации Cloudflare: 1000 файлов по 1 ГБ с 1000 просмотров каждого —
**$14.85/мес** на R2; на сервисе с платным egress это вышло бы в разы дороже.

## 7. Доставка контента пользователю в Telegram (решено)

Файлы не льются пачкой в чат — вместо этого приходит короткий отчёт + превью/ссылки на все файлы, с
пометками к каждому (для какой соцсети предназначен, тип контента, краткое описание, размер).
Пользователь может отдельно запросить прислать конкретный файл файлом или всю пачку целиком. Работает
одинаково с телефона и ПК (ссылки на R2, не зависят от устройства). Не засоряет чат и масштабируется
на любое количество файлов.

## 8. Риски и слабые места

### Технические риски
- Free-тариф Supabase (1 ГБ storage / 5 ГБ egress) — общий на все 4 агента; риск, если что-то по
  ошибке попадёт в Supabase Storage вместо R2 (например, черновик до принятия финального решения).
- R2 не настроен фактически — нужен bucket + API-токен на этапе разработки, аккаунт есть, но
  R2-специфичной настройки ещё не было.
- Контракт Агент 3 → Агент 4 требует доработки уже сданного (code-complete, все 10 слайсов) проекта
  Агента 3 — риск регресса в уже протестированном коде, нужны regression-тесты на новый узел
  (`handoffToAgent4`) по аналогии с тем, как это сделано у Агента 2.
- Midjourney (ключ в проекте есть) не имеет публичного self-serve API — не проектировать
  автоматизацию на него, только Runway/Ideogram/Imagen/Replicate.
- Стоимость генерации видео/изображений сильно варьируется по модели (от $0.015/сек до $0.70/сек) —
  без контроля бюджета один неудачный запуск каскада может быть дорогим.

### Продуктовые риски
- Публикация — новая ответственность, ранее не реализованная ни в одном из 4 агентов (в исходном
  3-агентном мастер-плане это была отдельная роль «Агент-Публикатор») — объём работы недооценён, если
  считать её "просто вызовом PostMyPost".
- Три режима в Агенте 1 — увеличивает сложность его диалогового UX, нужно бережно встроить в уже
  существующий flow, не сломав текущие 2 режима.
